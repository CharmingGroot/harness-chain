# 04. 서브에이전트 탭 (SubAgents)

## 개요

서브에이전트 탭은 특정 도메인 업무에 특화된 AI 에이전트를 정의하고 관리하는 화면이다. 서브에이전트는 오케스트레이터(부모 에이전트)에 의해 도구로 등록되어 병렬로 호출될 수 있다. AI 빌더를 통해 자연어 설명만으로 에이전트를 자동 설계할 수 있다.

---

## 서브에이전트 구조

```typescript
interface SubAgentDef {
  id: string;              // "sa_{timestamp}_{random4}"
  name: string;            // 에이전트 이름
  description: string;     // 한 줄 설명
  systemPrompt: string;    // 역할·분석 접근법·출력 형식 지시 (3~6문장)
  skills?: string;         // 할 수 있는 것들 (마크다운 불릿 4~7개)
  rules?: string;          // 제약·금지사항 (마크다운 불릿 3~5개)
  tools: string[];         // 사용 가능한 도구 이름 목록
  model: string;           // LLM 모델 ID
  maxIterations: number;   // 최대 반복 횟수 (기본 25)
  createdAt: string;
  updatedAt: string;
}
```

---

## 화면 구성

탭 내부는 두 가지 모드로 전환된다.

| 모드 | 설명 |
|---|---|
| 에이전트 목록 | 저장된 서브에이전트 카드 목록 |
| + 새로 만들기 | AI 빌더로 에이전트 생성 |

서브에이전트가 하나도 없으면 자동으로 빌드 모드로 시작한다.

---

## 에이전트 목록 (list 모드)

각 서브에이전트는 카드로 표시되며 다음 정보를 포함한다:
- 이름, 설명
- 시스템 프롬프트 앞 2줄
- 모델 배지 (claude 접두사 제거, 날짜 접미사 제거)
- 도구 배지 목록 (monospace)

---

## AI 빌더 (build 모드)

### 템플릿

빌드 화면 진입 시 3가지 기본 템플릿을 선택할 수 있다.

| 템플릿 | 설명 |
|---|---|
| 🔍 VIP 이탈 분석기 | VIP 등급 고객 이탈 위험 예측 |
| 🚨 이상 거래 탐지기 | 비정상 거래 패턴 감지 |
| 📊 데이터 리포터 | 경영진용 리포트 자동 작성 |

### 빌드 프로세스

1. 사용자가 자연어로 에이전트 역할을 설명한다.
2. `POST /api/subagents/build` 호출 (SSE).
3. GPT-5.4-mini가 에이전트 스펙을 JSON으로 생성한다.
4. SSE로 `{ type: "thinking" }` 이벤트 수신 후 `{ type: "result", data }` 수신.
5. 생성된 스펙이 편집 가능한 폼으로 표시된다.
6. 사용자가 수정 후 저장 버튼 클릭.

### /api/subagents/build 상세

```
POST /api/subagents/build
Content-Type: application/json
{
  description: string;           // 에이전트 역할 설명
  availableTools: {
    id: string; name: string; description: string;
  }[];                           // comingSoon 제외한 도구 목록
}

Response: text/event-stream
data: { type: "thinking", text: "서브에이전트 설계 중..." }
data: {
  type: "result",
  data: {
    name: string;
    description: string;
    systemPrompt: string;
    skills: string;          // 마크다운 불릿
    rules: string;           // 마크다운 불릿
    tools: string[];         // 도구 ID 배열
  }
}
```

LLM 프롬프트는 에이전트 역할 설명 + 사용 가능한 도구 목록을 제공하며, JSON만 응답하도록 지시한다. 응답에서 `/\{[\s\S]*\}/` 정규식으로 JSON 블록을 추출한다.

## 빌드 결과 편집 폼

AI가 생성한 결과는 즉시 저장되지 않고 편집 가능한 폼으로 표시된다.

| 필드 | 입력 타입 | 필수 |
|---|---|---|
| 이름 | text input | O |
| 설명 | text input | - |
| 시스템 프롬프트 | textarea (5행) | O |
| Skills 지침 | textarea (4행, monospace) | - |
| Rules (제약) | textarea (4행, monospace) | - |
| 도구 선택 | 토글 버튼 (다중 선택) | - |
| 모델 선택 | 토글 버튼 (Anthropic / OpenAI 그룹) | O |

도구 선택은 `comingSoon`이 아닌 도구만 표시된다. 모델은 `CHAT_MODELS` 배열을 제공자별로 그룹화하여 표시한다.

## 저장 (POST /api/subagents)

```
POST /api/subagents
{
  name, description, systemPrompt, skills, rules,
  tools: string[],
  model: string,
  maxIterations: 20
}
→ 201 Created: SubAgentDef
```

저장 후 목록 모드로 전환되고 `refreshAll()`로 레지스트리를 갱신한다.

---

## 도구 자동 선택 메커니즘

AI 빌더가 에이전트에 할당할 도구를 자동 선택하는 기준:

1. LLM 프롬프트에 사용 가능한 도구 목록(`id`, `name`, `description`)을 제공한다.
2. LLM이 에이전트 역할에 필요한 도구 ID 배열을 반환한다.
3. 반환된 도구 ID를 `selectedTools` Set으로 초기 적용한다.
4. 사용자가 빌드 결과 편집 폼에서 토글 버튼으로 수정할 수 있다.

## 오케스트레이터에서의 서브에이전트 사용

`Orchestrator.run()` 시 `listSubAgents()`로 저장된 서브에이전트를 모두 로드한다.

```typescript
for (const def of subAgentDefs) {
  // 서브에이전트 전용 도구 레지스트리 구성 (할당된 도구만)
  const subToolRegistry = new Registry<CATool>('tool');
  for (const toolName of def.tools) {
    const t = toolRegistry.get(toolName);
    if (t) subToolRegistry.register(toolName, t);
  }

  // SubAgentTool로 래핑하여 부모 레지스트리에 등록
  const subAgentTool = new SubAgentTool({
    name: `subagent_${def.id}`,
    description: `${def.name}: ${def.description}`,
    provider: resolveProvider(def.model, 4096),
    toolRegistry: subToolRegistry,
    systemPrompt: def.systemPrompt,
    maxIterations: def.maxIterations,
  });

  toolRegistry.register(subAgentTool.name, subAgentTool);
}
```

부모 AgentLoop는 서브에이전트를 일반 도구와 동일하게 호출한다. `@charming_groot/agent`의 `SubAgentTool`이 내부적으로 별도의 AgentLoop를 생성하여 위임 작업을 처리한다.
